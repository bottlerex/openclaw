# OpenClaw 性能基準與限制

**版本**: 1.0 (2026-03-05)
**基準日期**: 2026-03-05
**環境**: Mac mini (M1), Docker, 16GB RAM

---

## 性能指標

### 消息延遲 (End-to-End)

| 路徑 | P50 | P95 | P99 | 目標 |
|------|-----|-----|-----|------|
| Discord → 回應 | 1.2s | 1.8s | 2.5s | <2s |
| Slack → 回應 | 1.5s | 2.2s | 3.0s | <2s |
| Telegram → 回應 | 0.9s | 1.5s | 2.0s | <2s |
| WebChat → 回應 | 0.8s | 1.2s | 1.6s | <2s |

**延遲組成**:
```
消息進入 (50ms)
  → Bridge 路由 (30ms)
  → 代理處理 (800-1500ms)
    └─ RAG 搜索 (300-400ms)
    └─ Bedrock API 調用 (500-1000ms)
  → 回應發送 (50ms)
```

### 代理執行時間

| 操作 | 時間 | 備註 |
|------|------|------|
| 簡單回應 | 1-2s | 無 RAG，無工具 |
| RAG 搜索 + 回應 | 3-5s | 包含向量化 + 搜索 |
| 工具執行 + 回應 | 2-8s | 工具複雜度相關 |
| 代碼評審 | 8-15s | 調用 code-review agent |

### 並發能力

| 指標 | 當前 | 目標 | 限制因素 |
|------|------|------|---------|
| 同時會話 | ~50 | 100+ | 記憶體 + Bedrock 配額 |
| 每秒消息數 | ~10 | 50+ | Gateway 處理能力 |
| Telegram 消息率 | ~3/s | 30/s | Telegram API 限流 |
| 並發工具執行 | ~4 | 10+ | 沙箱資源 |

### 資源使用

**正常情況**:
```
CPU: 15-25% (4 核 M1)
內存: 600-800MB
磁盤: 1GB (logs) + 500MB (DB)
網絡: ~10 Mbps 峰值
```

**高負載情況**:
```
CPU: 50-70%
內存: 1.2-1.5GB
磁盤: I/O 激增
網絡: ~50 Mbps 峰值
```

**內存洩漏風險**:
- ⚠️ 長期運行 (30+ 天) 可能增長到 2GB
- 臨時對策: 每週重啟一次

## API 速率限制

### Bedrock (AWS)

| 限制 | 值 | 恢復 |
|------|-----|------|
| 請求/分鐘 | 10 | 自動恢復 |
| Token/分鐘 | 40,000 | 自動恢復 |

**超限行為**: 429 Too Many Requests → 指數退避重試

### Telegram

| 限制 | 值 | 備註 |
|------|-----|------|
| 消息/秒 | 30 | 群組可能更低 |
| 連接數 | 1 | 不支持多實例 |
| getUpdates 頻率 | 自動 | 長輪詢 |

**超限行為**: 429 Too Many Requests → 30 秒退避

### Brave Search

| 限制 | 值 | 恢復 |
|------|-----|------|
| 請求/天 | 2000 | 每天重置 |
| 請求/月 | 預付 | 無自動恢復 |

**超限行為**: 429 → 返回本地緩存結果

## 已知限制

### 系統級限制

| 限制 | 影響範圍 | 嚴重性 | 解決方案 |
|------|---------|--------|---------|
| **Gateway 單實例** | 單點故障 | 🔴 高 | Guardian auto-restart (臨時) / 多實例設計 (長期) |
| **執行超時 30s** | 長操作不支持 | 🟡 中 | 提交 async 任務 / 分解大操作 |
| **Token 限制 8K** | 長輸出不支持 | 🟡 中 | 分段生成 / 摘要輸出 |
| **工具白名單** | 新工具需手動添加 | 🟢 低 | 自動 allowlist 同期 (開發中) |

### 功能限制

| 功能 | 狀態 | 原因 | 替代方案 |
|------|------|------|---------|
| WhatsApp 完整支持 | ⚠️ Beta | API 審核中 | 使用 Telegram / Slack |
| RAG 圖像搜索 | ❌ 不支持 | 無多模態索引 | 使用 OCR 提取文本 |
| 實時協作編輯 | ❌ 不支持 | WebSocket 單向 | 生成代碼後客戶手動編輯 |
| 語音 I/O | ❌ 不支持 | 無音頻模型 | 客戶端側處理 |

### 通道限制

| 通道 | 限制 | 影響 |
|------|------|------|
| **Discord** | 消息大小 2000 字符 | 長輸出需分段 |
| **Slack** | 工作區隔離 | 無跨工作區操作 |
| **Telegram** | 30 消息/秒 | 批量操作需限流 |
| **LINE** | 推播限制 | 只能回應，不能主動推送 |

## P0 故障的性能影響

### Telegram Health-Monitor Stuck

**持續時間**: ~2 秒（每 30 分鐘）
**影響範圍**: Telegram bot 消息暫停
**用戶體驗**: 可能感知消息延遲

**修復前性能**:
```
可用性: 99.9% - 2秒/30分鐘 = ~99.89%
```

**修復後預期**:
```
自動恢復 < 5 秒
可用性: 99.99%
```

### Telegram 409 Conflict

**發生頻率**: 偶發 (每週 0-3 次)
**持續時間**: ~1-10 秒
**影響範圍**: Telegram bot 無法接收新消息

**修復後預期**:
- 0 次發生 (instance lock 防護)

### WebSocket 1006 Disconnect

**發生頻率**: 偶發 (每天 1-5 次)
**持續時間**: 5-30 秒 (自動重連)
**影響範圍**: WebChat UI 無法顯示

**修復後預期**:
- 自動重連時間 < 5 秒
- 無感知中斷

## 調優建議

1. **啟用結果緩存**
   ```json
   "cache": {
     "enabled": true,
     "ttl": 3600,
     "maxSize": "100MB"
   }
   ```

2. **RAG 索引優化**
   ```bash
   # 定期重建索引
   curl -X POST http://localhost:18789/rag/optimize
   ```

3. **限流配置**
   ```bash
   # 防止突發流量
   curl -X POST http://localhost:18789/rateLimit \
     -d '{"perSecond": 10, "burstSize": 20}'
   ```
